# No SPA — Server-Side Rendering Only

## Decision

The application uses **Jinja2 server-side rendering exclusively**. No SPA framework, no frontend build step, no npm.

## Context

track-finance is a single-user personal finance tracker. The UI complexity does not justify the overhead of a client-side framework. The app has:
- Standard CRUD forms
- Dashboard with Chart.js visualizations
- Filter/search interfaces
- No real-time collaboration or complex client state

## Rationale

- **Simplicity:** No `node_modules`, no webpack/vite, no frontend build pipeline. The entire frontend is Jinja2 templates + inline JS + Tailwind CDN.
- **Single user:** No concurrent editing, no optimistic updates, no offline-first requirements that would benefit from a SPA.
- **Deployment:** One Docker container serves everything. No separate frontend deployment.
- **Maintainability:** All rendering logic lives in Python templates. No context switching between frontend and backend codebases.

## Consequences

- All JavaScript is **inline in templates** — no separate `.js` files allowed.
- No npm, no `package.json`, no frontend build tools.
- Chart.js, Tailwind CSS, Lucide Icons, and fonts are loaded via **CDN only**.
- Page transitions are full server round-trips (acceptable for single-user).
- Dynamic UI interactions (chart drill-down, bulk edit) use inline `<script>` tags with `fetch()` for JSON API endpoints.

## Status

**Active and enforced.** This is a foundational decision documented in root `CLAUDE.md` under "Architecture Fundamentals" and "Do / Don't" sections.
